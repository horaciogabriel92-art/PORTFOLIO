Portfolio System Documentation

Complete technical documentation for the Horacio Morales portfolio platform. This system includes a public-facing portfolio, quote calculator, client onboarding system, and admin dashboard with real-time data management.

System Architecture

High-Level Overview

The system follows a modular architecture with clear separation between frontend, API, and data layers. Serverless functions handle backend logic, while Redis provides fast data access for real-time features.

Design Principles

  • Security First: All client PII is encrypted at rest using AES-256. JWT tokens for authentication.
  • Performance: Static site generation with Vite. Lazy loading for heavy components.
  • Scalability: Serverless architecture with Redis backend scales automatically.
  • Maintainability: Modular code structure with clear separation of concerns.

All API endpoints accept and return JSON. Timestamps are in ISO 8601 format. All requests must include proper Content-Type headers. 

# Request Headers

Authentication

The system uses JWT (JSON Web Tokens) for authentication. Tokens are signed with HS256 algorithm and expire after 1 hour. Protected endpoints require a valid token in the Authorization header.

Login Endpoint

POST /api/auth/login

Request Body

Field Type Required Description
password string Required Admin password (bcrypt hashed)

Response

{
  "success": true,
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "expires": "2026-03-08T14:30:00Z"
}

Token Verification

// Server-side verification middleware
import jwt from 'jsonwebtoken';

const verifyAuth = (req) => {
  const token = req.headers.authorization?.replace('Bearer ', '');
  
  if (!token) {
    throw new Error('Authentication required');
  }
  
  return jwt.verify(token, process.env.ADMIN_TOKEN);
};

Data Encryption

Client Personally Identifiable Information (PII) is encrypted before storage using AES-256-CBC encryption. This includes names, email addresses, and phone numbers.

Encryption Implementation

import crypto from 'crypto';

/**
 * Encrypts sensitive text data using AES-256-CBC
 * @param {string} text - Plain text to encrypt
 * @returns {string} Encrypted hex string
 */
export const encrypt = (text) => {
  const cipher = crypto.createCipher(
    'aes-256-cbc', 
    process.env.ENCRYPTION_KEY
  );
  let encrypted = cipher.update(text, 'utf8', 'hex');
  encrypted += cipher.final('hex');
  return encrypted;
};

/**
 * Decrypts encrypted data back to plain text
 * @param {string} encrypted - Encrypted hex string
 * @returns {string} Decrypted plain text
 */
export const decrypt = (encrypted) => {
  const decipher = crypto.createDecipher(
    'aes-256-cbc', 
    process.env.ENCRYPTION_KEY
  );
  let decrypted = decipher.update(encrypted, 'hex', 'utf8');
  decrypted += decipher.final('utf8');
  return decrypted;
};

Encrypted Fields

  • nombre_cliente - Client full name
  • email_cliente - Client email address
  • telefono - Client phone number
  • empresa - Company name (if provided)

Rate Limiting

To prevent abuse and ensure service availability, all API endpoints implement rate limiting based on client IP address.

Rate Limits

Endpoint Type Limit Window
Public Endpoints 100 requests Per hour per IP
Authentication 5 attempts Per 15 minutes per IP
Admin Operations 50 requests Per hour per IP

Implementation

// In-memory rate limiting (Redis recommended for production)
const rateLimit = new Map();

export const checkRateLimit = (identifier) => {
  const now = Date.now();
  const windowMs = 60 * 60 * 1000; // 1 hour
  const maxRequests = 100;
  
  if (!rateLimit.has(identifier)) {
    rateLimit.set(identifier, { count: 1, resetTime: now + windowMs });
    return { allowed: true };
  }
  
  const record = rateLimit.get(identifier);
  
  if (now > record.resetTime) {
    record.count = 1;
    record.resetTime = now + windowMs;
    return { allowed: true };
  }
  
  if (record.count >= maxRequests) {
    return { 
      allowed: false, 
      retryAfter: Math.ceil((record.resetTime - now) / 1000)
    };
  }
  
  record.count++;
  return { allowed: true };
};

Testing Strategy

Automated testing is currently minimal. Manual testing checklist below should be completed before each deployment.

Manual Testing Checklist

Authentication

  • Admin login with correct password succeeds
  • Admin login with wrong password fails
  • Token expires after 1 hour
  • Protected endpoints reject requests without token

Quote System

  • Quote form submission creates database entry
  • Client data is encrypted in database
  • Admin can view decrypted client data
  • Status updates persist correctly

Rate Limiting

  • Rate limit triggers after 100 requests
  • Rate limit resets after 1 hour
  • Different IPs have separate limits

Troubleshooting

Common Issues

Redis Connection Failed

Error: "Unable to connect to Redis" or timeout errors.

Solution: Verify KV_URL and KV_REST_API_TOKEN environment variables. Check if Upstash database is active in dashboard. Ensure IP whitelist includes Vercel IPs.

JWT Verification Failed

Error: "Invalid token" or "Authentication required" on protected endpoints.

Solution: Check ADMIN_TOKEN environment variable matches between login and verification. Token may be expired - re-authenticate.

Build Script Errors

Error: "ENOENT: no such file or directory" during build.

Solution: Ensure all directories (admin/, quote/, documentation/) exist. Check file permissions. Run build script from project root.

Trello Board Creation Fails

Error: Onboarding completes but no Trello board created.

Solution: Verify TRELLO_KEY and TRELLO_TOKEN. Check organization ID is correct. Ensure token has write permissions to organization.

Debug Mode

Enable verbose logging by setting environment variable:

DEBUG=true

This will log all API requests, database operations, and authentication attempts to the console.
